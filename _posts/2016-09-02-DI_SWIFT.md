## Storyboards and Segues와 함께하는 Swift View Controller 의 의존성 주입. (번역)

지난 주 [ 포스팅 이후](https://www.natashatherobot.com/i-heart-storyboards-nibs/) 엄청나게 다양한 질문들을 아래처럼 받았는데요.
`Q:custom initializer 없이 어떻게 View Controller 들 사이에 injecting dependencies 를 처리할 수 있나?` 그래서 이제 내가 사용하는 방법을 공유하려합니다.

첫째, 제 솔루션이 완벽하지는 않음을 언급하고 싶군요. 그리고 UIViewController 가 작동하는 방식에 따른 workaround 솔루션이라 볼 수 있습니다. 장차 애플에서 좀더 멋지고 swifty 한 UIKit API 를 제공했으면 하는 바램이 있습니다. 이때 물론 서브클래싱이 아닌 프로토콜을 이용하면 더 좋겠죠. 결국에는 dependency injection 을 UIKit 에서 매우 일상적인 수준에서 지원했으면 합니다.

둘째, 제가 복잡한 초기화를 싫어하는 이유는 UIKit 로 부터 UIViewController 를 서브클래싱해야 하기 때문입니다. 이미 여러개의 생성자가 존재하고 있고 super 를 기본적으로 호출해야 합니다. 왜냐면 UIKit 이 우리도 모르게 뒤쪽에서 뭔가 일을 꾸미고 있기 때문이죠.( 아시다시피 애플의 소스를 볼 수는 없죠) :

Swift

```
 
// comments from Apple's UIViewController code
/*
      The designated initializer. If you subclass UIViewController, you must call the super implementation of this
      method, even if you aren't using a NIB.  (As a convenience, the default init method will do this for you,
      and specify nil for both of this methods arguments.) In the specified NIB, the File's Owner proxy should
      have its class set to your view controller subclass, with the view outlet connected to the main view. If you
      invoke this method with a nil nib name, then this class' -loadView method will attempt to load a NIB whose
      name is the same as your view controller's class. If no such NIB in fact exists then you must either call
      -setView: before -view is invoked, or override the -loadView method to set up your views programatically.
    */
    public init(nibName nibNameOrNil: String?, bundle nibBundleOrNil: NSBundle?)
    public init?(coder aDecoder: NSCoder)
    
```
    
물론 한 동안 큰 변화는 없을 겁니다. 하지만 제가 UIKit 속을 들여다 볼 수 없는 관계로 똑똑한 자체적인 커스텀 생성자를 만들기 보다는 제 나름의 방법을 선호하게 되었는데요.

결론부터 말하자면 제가 이 문제를 해결하는 방식은 바로 Implicitly Unwrapped Optionals 를 사용하는 겁니다. 
 
이제 [이곳의 예시로](https://www.natashatherobot.com/protocol-oriented-segue-identifiers-swift/) 시작해 볼께요. 예제에 나오는 detail View controller 를 보시죠. 이 클래스는 이전 클래스의 특정 값(a value)에 의존적입니다.  detail View controller는 변수값에 대한 implicitly unwrapped optional 변수를 가지고 있는데. 이 변수는 초기화시점에 반드시 필요합니다.
 
Swift

```
// Detail View Controller
import UIKit
 
class RedPillViewController: UIViewController {
 
    @IBOutlet weak private var mainLabel: UILabel!
    
    // The Implicitly unwrapped optional!
    var mainText: String!
    
    override func viewDidLoad() {
        super.viewDidLoad()
 
        // it's used here!typ
        mainLabel.text = mainText
    }
 
}
```

보시다시피 여기서, mainText 값이 prepareForSegue에서 미리 할당되지 않았다면 여기 View Controller 에서 바로 크래시가 납니다. 바로 이게 핵심이고 문제를 쉽게 해결하는 방법입니다. 의도적으로 Crash 가 나도록 설계되어있고 따라서 바로 문제점이 파악이 되고 dependency를 할당시켜서 해결하면 됩니다.

Swift


```
// In the Parent / More General View Controller
    override func prepareForSegue(segue: UIStoryboardSegue, sender: AnyObject?) {
        
        switch segueIdentifierForSegue(segue) {
            
        case .TheRedPillExperience:
            let redPillVC = segue.destinationViewController as! RedPillViewController
            // assigning the implicitly unwrapped optional here!
            redPillVC.mainText = ""
        case .TheBluePillExperience:
            let bluePillVC = segue.destinationViewController as! BluePillViewController
            // assigning the implicitly unwrapped optional here!
            bluePillVC.mainText = ""
        }
    }
 ```   
핵심은 detail view controller의 초기화 또는 뷰로딩 싸이클과정에서 implicitly unused optional 값이 할당되거나 사용되어야 한다는 겁니다. 혹시라도 이 시점을 지난 이후에 사용되면 아래 문제점이 있기는 합니다만.
